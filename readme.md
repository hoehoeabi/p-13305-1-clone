## MVC 패턴
- Model: 애플리케이션의 데이터와 비즈니스 로직을 담당합니다. DB와 상호작용하며 상태를 관리합니다.

- View: 사용자에게 보여지는 UI(화면)입니다. 모델이 보유한 데이터를 표시합니다.

- Controller: 사용자의 요청을 해석하여 모델에 일을 시키고, 결과를 어떤 뷰에 보여줄지 결정하는 중개자입니다.

## 스프링에서 의존성 주입(DI) 방법 3가지 방법
DI는 객체를 직접 생성하지 않고 외부(스프링 컨테이너)로부터 주입받는 것입니다.

- 생성자 주입 (Constructor Injection)
  생성자를 통해 의존성을 주입받는 방식입니다.
  장점: 객체 생성 시점에 의존성이 확정되어 불변성을 보장하며, 테스트 코드 작성이 용이합니다.

- 수정자 주입 (Setter Injection)
  setXxx 메서드를 통해 의존성을 주입받습니다.
  주입받는 객체가 변경될 가능성이 있을 때 사용하지만, 런타임에 의존성이 바뀔 위험이 있어 자주 쓰이지 않습니다.

- 필드 주입 (Field Injection)
  변수 위에 @Autowired를 붙여 바로 주입받습니다.
  코드가 짧아 편하지만, 외부에서 수정이 불가능해 테스트가 어렵고 스프링 프레임워크에 너무 의존적이라 권장되지 않습니다.

## JPA의 장점과 단점
JPA는 자바 객체와 DB 테이블을 연결해주는 ORM(Object-Relational Mapping) 기술입니다.

- 장점:
  - 생산성: SQL을 직접 짜지 않아도 save(), find() 등으로 CRUD가 가능합니다.
  - 유지보수: 필드 하나 추가될 때 수많은 SQL을 일일이 수정할 필요가 없습니다.
  - 객체지향: DB 중심이 아닌 객체 중심의 설계를 가능하게 합니다.

- 단점:
  - 학습 곡선: 영속성 컨텍스트, 지연 로딩 등 개념이 다소 어렵습니다.
  - 성능 최적화: 복잡한 쿼리는 직접 짜는 것보다 성능이 떨어질 수 있으며, N+1 문제 같은 이슈를 잘 관리해야 합니다.
  - 결국 복잡한 쿼리는 직접 짜야합니다.

## HTTP GET 요청과 POST 요청의 차이
가장 큰 차이는 목적과 데이터의 위치입니다.
| 구분 | GET | POST |
| :--- | :--- | :--- |
| **목적** | 데이터 조회(Retrieve) | 데이터 생성/수정(Create/Update) |
| **데이터 위치** | URL의 쿼리 스트링 (예: `?id=1`) | HTTP Body 내부에 숨겨서 전송 |
| **캐싱** | 가능 (브라우저 기록에 남음) | 불가능 |
| **멱등성** | 멱등함 (여러 번 해도 결과가 같음) | 멱등하지 않음 |
